컴파일 버튼을 눌렀는데 PDF 대신 빨간 오류가 뜨면, 로그의 첫 줄부터 읽는 게 가장 빠릅니다. LaTeX 는 오류를 만난 자리에서 멈추고 *무엇이 이상했는지* 를 한 줄로 알려줍니다. 문제는 그 한 줄이 **에러가 감지된 자리** 이지 **에러가 시작된 자리** 가 아니라는 점입니다. 그래서 메시지를 그대로 검색해 봐도, 내 문서의 어느 줄을 고쳐야 하는지까지는 잘 나오지 않습니다.

Overleaf 를 쓴다면 화면 하단 **Logs** 탭에서 오류 전문을 볼 수 있고, 로컬에서 컴파일한다면 같은 내용이 `.log` 파일에 들어 있습니다. 이 글에서는 그 로그에 수식 때문에 가장 자주 찍히는 네 가지 메시지를 **원인 → 해결 → 오류 코드와 수정 코드 대비** 의 순서로 정리합니다.

| 오류 메시지 | 대개의 원인 |
|---|---|
| `! Missing $ inserted` | 수식 기호를 수식 환경 밖에서 씀 |
| `! Missing \endgroup inserted` | `{` 와 `}` 의 짝이 안 맞음 |
| `! Extra }, or forgotten $` | 수식 안에 `}` 가 하나 더 있거나 수식을 안 닫음 |
| `! Undefined control sequence.` | 명령어 철자가 틀렸거나 패키지가 없음 |

## 오류 1 — 수식 기호를 수식 환경 밖에서 쓴 경우

```
! Missing $ inserted.
<inserted text> $
```

**원인.** LaTeX 는 본문 모드와 수식 모드가 나뉘어 있고, `_` `^` `\frac` 같은 기호는 수식 모드 전용입니다. 본문에서 이 기호를 만나면 컴파일러가 *"여기서부터는 수식이어야 하는데 여는 기호가 없다"* 고 판단하고, 자기가 `$` 를 끼워 넣어 수습하려다 멈춥니다. 메시지의 `<inserted text> $` 가 그 *끼워 넣으려던* 문자입니다.

**해결.** 해당 기호를 `$...$` 또는 `\[...\]` 안으로 옮깁니다.

```latex
% 오류
속도 v_0 는...

% 수정
속도 $v_0$는...
```

본문 한복판보다 더 자주 걸리는 자리가 따로 있습니다. 표 셀 안, 그림 캡션 `\caption{}`, 각주 `\footnote{}` 처럼 *본문처럼 안 보이는 본문* 입니다. 변수 이름 하나만 짧게 적고 지나가는 자리라 여는 기호를 빼먹기 쉽습니다.

같은 메시지가 전혀 다른 이유로 나오기도 합니다. 수식을 열고 닫지 않았을 때, 그리고 `equation` · `align` 환경 안에 빈 줄을 넣었을 때입니다. 빈 줄 쪽은 <a href="/blog/posts/latex-equation-5-mistakes/" data-proofer-ignore>LaTeX 수식에서 흔히 하는 5가지 실수</a> 의 마지막 항목에서 코드와 함께 다뤘습니다.

## 오류 2 — 중괄호 짝이 맞지 않는 경우

```
! Missing \endgroup inserted
```

**원인.** `{` 와 `}` 의 짝이 맞지 않습니다. 여는 중괄호가 닫히지 않은 채로 그룹이 끝나야 할 지점에 도달하면 컴파일러가 멈춥니다.

**해결.** 중괄호를 하나씩 세어 짝을 확인합니다. 눈으로 셀 때는 인자를 받는 명령어부터 봅니다 — `\frac{}{}` 는 두 개, `\sqrt{}` `\text{}` `\mathbb{}` 는 각각 한 개입니다. 중첩이 깊어질수록 여는 쪽만 늘어나기 쉽습니다.

```latex
% 오류 — 여는 중괄호 3개, 닫는 중괄호 2개
$\frac{\sqrt{x+1}{2}$

% 수정
$\frac{\sqrt{x+1}}{2}$
```

세는 일은 에디터에 맡기는 편이 빠릅니다. Overleaf 를 포함한 대부분의 LaTeX 에디터는 커서를 괄호에 대면 짝이 되는 괄호를 함께 강조해 줍니다. 짝이 없는 괄호는 강조가 뜨지 않으니, 의심되는 줄에서 커서를 옮겨 보면 어느 쪽이 비었는지 금방 드러납니다.

## 오류 3 — 닫는 중괄호가 하나 더 있거나, 수식을 닫지 않은 경우

```
! Extra }, or forgotten $
```

**원인.** 오류 2 의 반대쪽입니다. 수식 안에서 `}` 가 하나 더 있거나, 수식을 닫지 않은 채로 끝냈습니다.

**해결.** 수식 환경의 시작과 끝을 확인합니다.

```latex
% 오류 — 닫는 중괄호가 하나 더
$\frac{a}{b}}$

% 수정
$\frac{a}{b}$
```

메시지가 `}` 와 `$` 를 한 문장에 함께 언급하는 데는 이유가 있습니다. 컴파일러 입장에서는 두 상황이 같은 증상으로 보이기 때문입니다. *열린 그룹이 정리되기 전에 수식이 끝나 버린 자리* 를 만났을 뿐이고, 그게 여분의 중괄호 때문인지 빠뜨린 수식 기호 때문인지는 문서를 봐야 압니다. 그래서 이 메시지가 뜨면 중괄호와 수식 기호를 **둘 다** 확인하는 게 맞습니다.

## 오류 4 — 명령어를 알아보지 못하는 경우

```
! Undefined control sequence.
l.45 \frac
```

**원인.** 철자가 틀린 명령어를 입력했거나, 그 명령어를 정의하는 패키지가 프리앰블에 없습니다.

**해결 1 — 철자 확인.** LaTeX 명령어는 대소문자를 구분합니다. `\Frac` 은 `\frac` 이 아니고, `\Alpha` 는 아예 존재하지 않습니다. 로그의 `l.45` 뒤에 문제가 된 명령어 이름이 그대로 찍히기 때문에, 네 가지 중에서는 위치를 특정하기가 가장 쉬운 오류입니다.

**해결 2 — 패키지 확인.** 철자가 맞는데도 같은 메시지가 뜬다면 패키지 누락입니다. 프리앰블에 추가합니다.

```latex
\usepackage{amsmath}
\usepackage{amssymb}
```

`\text{}` `\operatorname{}` 과 `align` 환경은 `amsmath` 에, `\mathbb{}` `\mathfrak{}` 같은 글꼴 명령어는 `amssymb` 에 들어 있습니다. 수식을 쓰는 문서라면 이 두 줄은 처음부터 넣어 두는 편이 낫습니다.

## 로그를 읽는 순서

네 가지를 다 알아도, 오류가 한 번에 여러 개 뜨면 어디부터 손대야 할지 막막합니다. 순서는 단순합니다.

1. **첫 번째 오류부터 고칩니다.** 앞의 오류가 파서를 어긋난 상태로 밀어 놓기 때문에, 뒤의 오류 대부분은 첫 오류의 그림자입니다. 하나 고치고 다시 컴파일하면 열 개가 한꺼번에 사라지는 일이 흔합니다.
2. **`l.N` 은 출발점이지 목적지가 아닙니다.** 그 줄에서 아무것도 안 보이면 그 위쪽 문단을 봅니다.
3. **직전에 고친 자리를 먼저 의심합니다.** 어제까지 컴파일되던 문서라면 원인은 오늘 새로 쓴 줄에 있습니다.

## 정리

| 오류 메시지 | 확인할 것 |
|---|---|
| `! Missing $ inserted` | 본문에 나온 `_` `^` `\frac` 을 수식 안으로. 캡션 · 표 셀 · 각주도 확인 |
| `! Missing \endgroup inserted` | 여는 중괄호가 닫혔는지. 인자를 받는 명령어부터 |
| `! Extra }, or forgotten $` | 여분의 `}` 와 닫지 않은 수식, 둘 다 |
| `! Undefined control sequence.` | 명령어 철자 (대소문자), 그리고 `amsmath` · `amssymb` |

메시지 자체는 네 개뿐이고, 원인도 결국 *모드를 벗어났거나, 짝이 안 맞거나, 이름을 못 찾거나* 셋 중 하나입니다. 로그의 첫 줄을 이 세 갈래로 분류하는 습관이 붙으면 디버깅 시간이 눈에 띄게 줄어듭니다.
